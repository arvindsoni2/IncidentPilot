# IncidentPilot v0.2 Milestone 2: Persistence and Lifecycle

## Objective

Make existing local databases safely upgradeable and ensure every analysis and
evaluation outcome is queryable without expanding IncidentPilot's read-only
safety boundary.

## Delivered scope

- Alembic migrations with an MVP baseline and v0.2 schema revision
- safe adoption of unversioned MVP databases
- fresh-database and upgrade/downgrade migration tests
- explicit `failed` incident state with failed agent-run details
- persisted evaluation runs and individual check results
- evaluation history list, JSON export, and confirmed age-based pruning
- documented retention behavior for evidence, reports, and evaluations

## Compatibility

`incidentpilot db init` remains the operator entry point. It now upgrades to the
latest migration revision. When it finds an unversioned database containing the
MVP schema, it stamps the baseline revision and applies only later migrations,
preserving existing records.

## Safety boundaries

This milestone does not add remediation, arbitrary commands, authentication,
live scenario automation, or automatic incident-data deletion.

## Acceptance criteria

1. Fresh databases upgrade to migration head.
2. An unversioned MVP database upgrades without losing records.
3. Upgrade/downgrade smoke tests pass.
4. Analysis exceptions leave incidents in `failed`, not `analyzing`.
5. Eval runs and checks are queryable from the database.
6. Eval history can be listed, exported, and deliberately pruned.
7. Incident evidence and reports have a documented no-automatic-deletion rule.
8. Existing tests, deterministic evals, clean-checkout smoke, and Compose build
   remain green.
