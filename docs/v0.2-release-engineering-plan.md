# IncidentPilot v0.2 Milestone 1: Release Engineering Plan

## Purpose

IncidentPilot MVP is already working. This milestone hardens the project so it is credible for a technical portfolio/interview reviewer first, and a potential client/employer demo second.

The focus is **release confidence**, not new product behaviour.

## Source-of-truth inputs

This plan is based on:

- `MVP_GAPS.md`
- `NEXT_ITERATION_PLAN.md`
- Repository: `https://github.com/arvindsoni2/IncidentPilot`
- Current owner decisions captured during v0.2 clarification

## Milestone positioning

v0.2 is planned as a balanced iteration covering:

1. Release engineering
2. Persistence/lifecycle
3. Integration confidence
4. Local product polish

This file covers only:

> **v0.2 Milestone 1: Release Engineering**

## Audience

### Primary

Technical portfolio/interview reviewer.

### Secondary

Potential client/employer demo audience.

This means the project should be easy to clone, verify, and understand from the repository.

## Goals

Milestone 1 should make this true:

> A reviewer can clone IncidentPilot, install dependencies reproducibly, run quality checks, run deterministic tests/evals, validate Compose configuration, build images, and understand the local demo path.

## Non-goals

This milestone must not implement or design product expansion beyond release engineering.

Explicitly out of scope:

- remediation/self-healing execution
- action executors
- approval workflows
- arbitrary shell execution
- database migrations
- incident lifecycle/schema changes
- eval persistence
- authentication/authorization/CSRF
- UI redesign
- live FS-001/FS-002 scenario automation
- Docker/Podman first-class parity
- broad application refactors
- screenshots/GIFs

Manual recovery recommendations in reports are acceptable for v0.2 product direction, but they are not part of this Milestone 1 implementation unless already present.

## Confirmed decisions

| Area | Decision | Rationale |
|---|---|---|
| Milestone name | `v0.2 Milestone 1: Release Engineering` | Keeps this work separate from migrations, live integration, and product polish. |
| Dependency management | `uv` with committed `uv.lock` | Provides reproducible dependency setup and reviewer-friendly install flow. |
| Packaging | Clean `pyproject.toml` pass | Modernizes setup without a full tooling overhaul. |
| Dependency groups | Main dependencies + one `dev` group | Simple reviewer path: `uv sync --group dev`. |
| Python support | Python 3.11 and 3.12 | Stable modern baseline without chasing newer plugin warnings. |
| CI platform | GitHub Actions | Visible and familiar for repo reviewers. |
| Workflow file | `.github/workflows/ci.yml` | Simple conventional name. |
| CI jobs | `python-quality` and `compose-validation` | Avoids rebuilding Compose images for each Python version. |
| CI trigger | Push/PR for `main` plus `workflow_dispatch` | Common default plus manual rerun option. |
| CI permissions | Explicit read-only, e.g. `contents: read` | Safer and reviewer-friendly. |
| uv setup in CI | `astral-sh/setup-uv` | Standard GitHub Actions path for uv. |
| uv cache | Enabled where suitable | Speeds CI without changing product behaviour. |
| CI dependency sync | Locked/frozen sync | CI must fail if `pyproject.toml` and `uv.lock` drift. |
| Linter | Ruff only | Fast, modern, simple. |
| Ruff strictness | Conservative | Avoids large style refactor. |
| Ruff enforcement | Fail CI immediately | Gives clear quality signal. |
| Ruff config | Small explicit rule set | Predictable conservative linting. |
| Ruff target version | Python 3.11 | Matches minimum supported Python. |
| Ruff line length | Inspect existing repo style first | Avoid unnecessary churn. |
| Ruff formatting | Not included in Milestone 1 | Keeps scope conservative. |
| Ruff auto-fix | Safe fixes allowed | Codex may use safe auto-fixes and report diff. |
| Type checking | Not included unless already mature | Avoids tooling rabbit hole. |
| Makefile | Add local parity entrypoint | Makes CI checks reproducible locally. |
| Main local targets | `install`, `lint`, `test`, `eval`, `check`, `verify`, `compose-check`, `compose-build`, `ci-local` | Short, conventional, easy to document. |
| `make install` | `uv sync --group dev` | Simple predictable install target. |
| `make lint` | Ruff lint only | Conservative quality check. |
| `make test` | Normal pytest suite only | Keeps evals explicit. |
| `make eval` | Existing deterministic eval mechanism only | Do not invent eval behaviour. |
| `make check` | lint + normal tests | Fast developer loop. |
| `make verify` | lint + normal tests + deterministic evals | Docker-free product correctness loop. |
| `make ci-local` | full CI parity | Includes `verify`, Compose config, and Compose build. |
| Docker local behaviour | `ci-local` fails if Docker/Compose unavailable | Keeps command semantics clean. |
| Compose command | `COMPOSE ?= docker compose -f infra/compose.yaml` override | Docker primary, Podman best-effort locally. |
| Compose CI scope | Config validation + image build | Live scenario automation is deferred. |
| Runtime positioning | Docker primary, Podman best-effort locally | Matches reviewer/CI path and Fedora local reality. |
| README strategy | Reviewer quickstart | README should be easier to scan. |
| Developer guide | `docs/developer-guide.md` | More detailed local setup and command reference. |
| README badge | Add GitHub Actions CI badge | Immediate reviewer confidence signal. |
| Screenshots/GIFs | Not in Milestone 1 | Visual polish comes later. |
| Planning docs | Create human plan + Codex prompt file | Separates intent from execution. |
| Prompt strategy | Hybrid master prompt + sub-prompts | Resumable and quota-friendly. |
| First Codex pass | Inspect-only + baseline checks | Establish before/after state. |
| Baseline failures | Stop, report, classify; no edits | Prevents mixing discovery and remediation. |
| Branch handling | Codex may create feature branch | Only if working tree is clean. |
| Commit handling | Codex may commit after checks pass | No commit if verification fails. |
| Push handling | Codex may push branch | Only after remote is confirmed. |
| PR handling | Codex may create draft PR | Use GitHub CLI if available/authenticated. |
| PR scope boundary | Required | Reviewers should understand this is release engineering only. |

## Proposed Makefile target semantics

```makefile
COMPOSE ?= docker compose -f infra/compose.yaml

install:
	uv sync --group dev

lint:
	uv run ruff check .

test:
	uv run pytest

eval:
	# Must call the existing deterministic eval mechanism.
	# Codex must inspect and use the real command.

check: lint test

verify: lint test eval

compose-check:
	$(COMPOSE) config --quiet

compose-build:
	$(COMPOSE) build

ci-local: verify compose-check compose-build
```

The exact eval/test/module commands must be confirmed from the current repository.

## CI structure

### Workflow

`.github/workflows/ci.yml`

### Triggers

- push to `main`
- pull request to `main`
- manual `workflow_dispatch`

### Permissions

```yaml
permissions:
  contents: read
```

### Job 1: `python-quality`

Matrix:

- Python 3.11
- Python 3.12

Responsibilities:

- checkout
- set up Python
- set up uv
- restore/use uv cache where suitable
- locked/frozen dependency sync
- Ruff lint
- normal tests
- deterministic evals

### Job 2: `compose-validation`

Runs independently in parallel.

Responsibilities:

- checkout
- validate Compose configuration
- build Compose images
- do not start live failure scenarios

## Acceptance criteria

Milestone 1 is complete when:

1. `uv.lock` is generated and committed.
2. `uv sync --group dev` works from a fresh checkout.
3. `make install` works.
4. `make lint` runs conservative Ruff linting.
5. `make test` runs the normal test suite.
6. `make eval` runs the existing deterministic eval suite.
7. `make check` runs lint + normal tests.
8. `make verify` runs lint + normal tests + deterministic evals.
9. `make compose-check` validates Compose configuration.
10. `make compose-build` builds Compose images.
11. `make ci-local` runs the full local CI-parity check and fails clearly if Docker/Compose is unavailable.
12. GitHub Actions runs `python-quality` across Python 3.11 and 3.12.
13. GitHub Actions runs `compose-validation` independently.
14. GitHub Actions uses locked/frozen dependency sync.
15. CI passes on GitHub.
16. README has a reviewer-friendly quickstart and CI badge.
17. `docs/developer-guide.md` exists and documents local development.
18. Strong scope boundaries are preserved: no remediation, no migrations, no live scenario automation, no auth, no broad refactor.

## Implementation sequence

1. Inspect repository and run baseline checks without editing.
2. Migrate packaging to uv and generate lock file.
3. Add conservative Ruff config and fix only necessary lint issues.
4. Add Makefile targets.
5. Add GitHub Actions CI.
6. Update README and create developer guide.
7. Run verification locally.
8. Commit, push branch, and create draft PR if all checks pass.
